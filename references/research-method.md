# 调研方法论总则

本 skill 只做一件事:为后续策展大纲准备一份**足够深、足够全、足够新、可核验、可直接打开的知识型 XMind 资料导图**。用户明确要求报告、Word、docx 或完整交付时,再基于同一批资料追加一份资料报告型 Word。

默认最终成果不是策展方案,不是展示建议。默认只交 `.xmind`;完整交付模式追加的 `.docx` 也是资料报告。

## 总原则

正式调研必须遵循“知识系统先行,证据校准托底”:

1. 先建立主题知识系统:主题是什么,不是什么,怎么分类,由哪些对象构成,如何运作,谁参与,在哪里发生,有什么历史、数据、规则、争议和最新变化。
2. 再用来源校准硬事实:年份、人物、地点、数据、法规、机构、新闻、财务、馆藏、赛事、政策等不能靠模型猜。
3. 证据卡是原料,不是最终阅读形状。最终 XMind 要像给完全不懂的人讲知识,而不是把证据字段机械展开。
4. 百科、教材、综述、入门手册负责建立知识入口和术语地图;官方、学术、监管、数据库、主流媒体负责校准事实。
5. 政府、监管、法院、交易所、官方数据库、robots 禁止或反爬页面不得用脚本抓取;只能人工/浏览器读取后用 `manual_source_note.py` 记录少量必要摘录。

## 生产链路

1. 主题确认:类型、地域、时间范围。
2. 检索计划:`research_plan.py` 生成 `research_plan.json`。
3. 搜索收集:`search_collect.py` 搜索或导出查询任务。
4. 来源获取:`fetch_sources.py` 获取普通公开网页;受限或官方来源转 `manual_required`。
5. 人工来源记录:`manual_source_note.py` 记录浏览器/人工读取的公开摘录。
6. 证据卡:从来源中抽取 `evidence_cards.jsonl`。
7. 证据校验:`evidence_cards.py validate`。
8. 覆盖审计:`coverage_audit.py`。
9. 深搜循环:`research_loop.py` 根据覆盖缺口生成补搜任务。
10. 模型解释层:补足概念解释、层级拆分、术语关系和报告衔接。
11. 导图大纲:覆盖通过后写 `调研大纲.md`。
12. 大纲审计:`outline_audit.py`。
13. 生成 XMind:`md_to_xmind.py`。
14. 最终审计:`xmind_audit.py`。
15. 完整交付模式:报告 Markdown、报告审计、生成 Word。

任何一步失败,都回到上一步补资料,不要用占位句糊过去。

## 来源合规

`fetch_sources.py` 只用于普通公开网页。它必须尊重 robots.txt,不得绕过登录、付费、验证码、反爬或访问控制。

以下来源默认人工处理:

- 政府网站、监管机构、法院、交易所、官方数据库。
- robots.txt 不允许自动访问的页面。
- 访问条件不清、反爬明显或需要人工阅读上下文的页面。

人工处理不是放弃来源。正确做法是:

```bash
python3 "<skill>/scripts/manual_source_note.py" \
  --out "03_来源文本/sources.jsonl" \
  --append \
  --url "公开页面 URL" \
  --title "来源标题" \
  --excerpt "人工阅读后整理的必要摘录" \
  --source-type government
```

记录时只写必要摘录或自己的概括,不要整篇复制。后续证据卡仍要写清来源标题、URL、来源性质和采信边界。

## 百科与模型知识的使用

百科类来源包括百度百科、维基百科、机构百科、专业词典、教材概览和入门手册。它们优先用于:

- 名称、别名、简称、英文名、旧称。
- 上位概念、下位概念、相邻概念。
- 分类入口、术语表、常见误区。
- 主题历史和代表对象的初步索引。

百科不能单独支撑高风险事实,例如精确数字、最新动态、政策条款、财务数据、法律状态和争议结论。这类事实要回到官方、学术、数据库、主流媒体或多来源交叉。

模型知识可以用于:

- 把概念讲给小白听。
- 解释工具、机制、流程、术语差异。
- 建立章节顺序、概念关系和过渡段。
- 从碎片资料中识别缺口和继续搜索方向。

模型知识不能直接生成:

- 具体年份、人物关系、地点、法规条款、市场数字。
- 最新新闻、监管状态、来源标题、URL。
- 未经来源支持的行业判断、历史评价或文化结论。

模型提出的新硬事实,必须先进入补搜任务或待核实清单,找到来源后再进入证据卡。

## 深搜循环

覆盖审计失败时,不要凭感觉补几条搜索。必须运行:

```bash
python3 "<skill>/scripts/research_loop.py" \
  --plan "01_检索计划/research_plan.json" \
  --coverage "05_覆盖审计/coverage_audit.json" \
  --evidence "04_证据卡/evidence_cards.jsonl" \
  --out "01_检索计划/research_loop.json" \
  --breadth 4 \
  --depth 2
```

逐项阅读 `research_goal`、`expected_evidence_fields`、`stop_condition`、`visited_urls`。补搜必须围绕缺口核心展开,不能重新泛搜主题词。新来源进入 `sources.jsonl`,新事实进入 `evidence_cards.jsonl`,再重新跑校验和覆盖审计。

## 资料价值观

资料调研的价值不在于几句观点,而在于保留后续大纲可调用的知识材料:

- 定义、边界、上下位概念。
- 历史阶段、时间线、事件过程。
- 人物、组织、群体、关系网络。
- 地理空间、区域差异、场景条件。
- 分类谱系、对象、器物、作品、文本、档案。
- 原理、流程、工艺、操作方法。
- 数据、规模、价格、市场、资本、政策。
- 社会文化、语言符号、媒介传播。
- 学术文献、权威来源、争议和待核实项。

每条重要资料都尽量回答:谁、何时、何地、发生什么、数据多少、来源在哪、来源可信度如何、是否有不同说法。

## 从后续大纲反推调研深度

完整策展大纲会展开到篇章、单元、小节、具体知识点、人物、事件、器物、地点、作品、工艺、政策、数据、说明文字和资料来源。因此前期资料导图必须比大纲更细。

反推检查:

- 每个一级资料分支至少能反推出 2-3 个后续大纲小节。
- 每个重要概念至少能支撑一段 150-300 字正文。
- 每个资料组至少包含对象、事件、人物、时间、地点、数据、来源中的 3 类。
- 如果一个分支只能写出口号、情绪判断或空泛概括,说明资料密度不足。

## Word 资料报告定位

Word 报告不是把导图节点复制成文档,而是把同一批资料转成可阅读、可追溯、层级清楚的资料文本。

正确的 Word 报告应当具备:

- 总述:说明主题边界、资料范围、来源基础和阅读方式。
- 章节:每章回答一个资料问题,如定义、历史、人物、对象、机制、空间、数据、监管、同类对照。
- 段落:把多条证据卡合并成连续说明,句子有长短变化,不是短句清单。
- 来源口径:说明哪些资料来自官方、监管、学术、媒体、数据库或估算。
- 待核实问题:明确哪些数据会变化,哪些来源需要后续跟进。

Word 报告不得出现字段腔,例如“时间口径为”“相关人物包括”“涉及对象包括”“数据口径包括”“这些信息分别见于”。正确写法是先用自然句交代对象和事实,再补足身份、时间、地点、数据或来源。来源句放在段末即可,例如“以上信息综合参考:来源 A、来源 B。”

## 内容丰富不是分支越多

失败的大纲常见三种:

- 分支很多,但每个分支只有短词或一句话。
- 分支很多,但内容是内部待办。
- 分支很多,但内容是方法标签,没有真正增加事实。

正确标准:

- 每个重要概念都有多层解释。
- 每个关键事实能追到来源。
- 每个分类名词都有定义、边界、典型特征、使用或出现条件、来源。
- 每个数字有年份、单位、口径和来源。
- 每个人物有身份、关系、事件、时间、来源。
- 每个作品、器物或档案有名称、年代、作者或相关人物、内容或形制、流传或收藏、来源。

## 证据卡要求

证据卡是事实校准层。每条证据卡至少包含:

- `claim`:完整事实陈述,不能只写短词。
- `core`:对应 `research_plan.json` 的必查核心。
- `dimension`:对应搜索维度。
- `time`:时间或年代。
- `people`:人物。
- `places`:地点。
- `objects`:作品、器物、事件、机构、产品、政策等对象。
- `data`:数字、金额、规模、参数、口径。
- `source_title`:来源标题。
- `source_url`:来源 URL 或用户资料文件名。
- `source_type`:government / regulatory / court / exchange / academic / institution_or_database / media / web / user 等。
- `confidence`:官方确认 / 监管文件 / 学术资料 / 机构资料 / 媒体报道 / 行业资料 / 分析师估算 / 单一来源 / 待核实。

不能把 `evidence_seed.jsonl` 当证据卡。每条证据卡至少填实 `time`、`people`、`places`、`objects`、`data` 中的 2 个字段。字段为空、写“不详/待补/正式使用时要”都不能算有效证据。

可选解释字段:

- `explanation`:给小白看的解释。
- `mechanism`:机制、原理或流程。
- `boundary`:边界和相邻概念差异。
- `misconception`:常见误区。
- `teaching_points`:可以进入导图的讲解要点。

## 禁止越界

不要写展示形式、展项创意、互动方式、空间表达、展线设计、策展主题、核心张力、独特价值、主题口号、同类展馆可借鉴点、展陈启示、观众体验建议。

要写资料事实是什么、来源在哪里、发生在什么时候、涉及谁、数据是多少、谁这么说、是否官方确认、还有哪些不同说法。

## 搜新标准

现代企业、机构、政策、科技、产业、人物、消费、社交传播主题必须查当前年份、近一年、近 90 天,必要时近 30 天。传统历史文化主题也必须查最新研究、出版、保护、数字化、公共传播和教育活动。

时效信息必须写清发布时间、事件发生时间、来源状态和是否可能变化。

## 污染判定

最终 `.xmind` 和 `.docx` 不得出现:

- 内部待办句,例如“正式使用时要”“需要继续”“需要补”“待补”。
- 方法标签,例如“继续下钻方向”“下钻方式”“关联资料面”“概念资料卡”“来源类型为”“可信度标注为”“对应证据卡”。
- 字段腔,例如“时间口径为”“相关人物包括”“相关地点包括”“涉及对象包括”“数据口径包括”“这些信息分别见于”。
- 策展污染,例如“展项”“展示形式”“空间表达”“策展主题”“核心张力”“可借鉴”“观众体验”。

正确处理:已有资料直接写事实、数据、时间、地点、人物、来源;暂无资料写清已查来源和缺口对象,不把内部检查表、搜索任务、质量提醒变成导图节点。
